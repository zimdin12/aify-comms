What is this?

This is the aify-comms control plane: a dashboard for connected Windows, WSL, Linux, Docker, and remote bridges. Use it to spawn managed agents into chosen workspaces, chat with them, inspect runs, stop/restart sessions, and keep old manual CLI bindings visible only when they matter.

Quick Setup

Fast path for Claude Code:

cd /path/to/aify-comms
bash install.sh --client claude http://localhost:8800 --with-hook

Fast path for Codex:

cd /path/to/aify-comms
bash install.sh --client codex http://localhost:8800 --with-hook

Fast path for OpenCode:

cd /path/to/aify-comms
bash install.sh --client opencode http://localhost:8800

Restart the client after install. Then run an environment bridge in every execution environment you want the dashboard to use:

cd /path/to/workspace-or-workspace-parent
aify-comms

On native Windows PowerShell/cmd, run aify-comms.cmd. The service URL defaults to http://localhost:8800. The current directory is always an allowed workspace root; extra root arguments are optional safety boundaries. The actual project folder is selected per spawned agent in Environments → Spawn Agent.

Manual comms_register remains useful for human-open resident CLI sessions, but it is not the normal dashboard spawn path. Active triggering works through the local stdio MCP bridge on the target machine. SSE clients can use the shared service but cannot launch local Claude/Codex/OpenCode work themselves.

Codex CLI note: if you want visible live wakeups instead of background thread resume, start Codex through codex-aify before registering. From that live session, prefer the strongest exact registration first: comms_register(..., runtime="codex", sessionHandle="$CODEX_THREAD_ID", appServerUrl="$AIFY_CODEX_APP_SERVER_URL"). Plain codex still works, but it advertises the older codex-thread-resume fallback.

Claude CLI note: resident wakeups require starting Claude through the installed claude-aify wrapper so the local aify channel bridge is allowed to load during the current research-preview channels phase. It is only installed when the Claude client was set up with a real shared aify server URL.

Manual vs managed: comms_register binds a live resident CLI session you opened yourself. Those rows are shown as manual CLI bindings, and offline entries are historical identities, not killable processes. Dashboard-spawned managed agents are the normal control-plane path: they have environment, workspace, runtime, spawn spec, session state, and recovery metadata. Normal chat should use comms_send(...) or the Chat page.

Live send: normal messages are not queued for future runs. If the target is offline, stale, stopped, already working, already has queued work, or lacks a live wake path, Send reports the issue and writes no message. Agent-reported blocked state and legacy completed state are status notes, not delivery blockers.

Dashboard composer: Send is the normal teamwork path. It delivers only to currently startable agents and surfaces live delivery failures immediately.

Core Concepts

  • Chat: conversation history and composer. Use Viewing as to read and send from a specific identity, such as manager. Read/unread and delete actions are scoped to the selected identity or selected conversation.
  • Channel: shared room inside Chat. Use it for group broadcasts and team history. It is not a separate top-level workflow.
  • Agent: teammate identity. Managed agents are dashboard-spawned and recoverable. Manual/resident CLI bindings are human-open compatibility sessions.
  • Session: concrete runtime backing record for an agent: environment, workspace, process/session handle, resume capability, and lifecycle state.
  • Run: tracked execution attempt attached to a message. Use it when you need progress, events, interrupt/steer, failure state, or required handoff tracking.
  • Artifact: shared file or text blob. Chat can upload a file and insert its reference; the Artifacts page lists and deletes stored artifacts.

Continue / Compact

Open Sessions and use Continue as. The dashboard opens an editable continuation form with new agent ID, role, environment, runtime, workspace, and a generated packet. The packet includes the last selected number of direct messages involving the source agent plus channel posts authored by it, then queues a fresh managed-warm spawn. This is useful for compaction, but keep the included message count bounded so the new session starts cleanly.

Tools

  • Messaging: comms_register, comms_envs, comms_spawn, comms_agents, comms_status, comms_agent_info, comms_send (normal teamwork; live-gated), comms_listen (stdio only), comms_inbox, comms_unsend, comms_search
  • Advanced run control: comms_dispatch (debug/strict run tracking), comms_run_status, comms_run_interrupt. Normal users and agents should send messages, not dispatches.
  • Channels: comms_channel_create, comms_channel_join (+ invite others), comms_channel_send (live-gated), comms_channel_read, comms_channel_list. In the dashboard, channels live inside Chat.
  • File sharing: comms_share (text + binary), comms_read, comms_files
  • Management: comms_clear, comms_dashboard

Dashboard Settings

These settings are still relevant, but they are admin/maintenance settings: retention, file cleanup, presence labels, stale identity rotation, and browser refresh. They do not configure runtime models or spawn profiles yet.

API Endpoints

Mostly useful for debugging integrations and tests; daily work should happen through Home, Chat, Team, Environments, Sessions, Runs, and Artifacts.

  • GET /health — Health check
  • GET /mcp/sse — MCP SSE endpoint
  • GET/POST/DELETE /api/v1/agents — Agents
  • POST /api/v1/agents/{id}/control — Stop, resume, or interrupt active work for an identity
  • GET /api/v1/environments — Connected execution environments
  • POST /api/v1/environments/heartbeat — Environment bridge heartbeat
  • GET/POST /api/v1/spawn-requests — Dashboard spawn queue
  • POST /api/v1/spawn-requests/claim — Bridge claim next spawn for an environment
  • GET /api/v1/sessions — Runtime session records
  • POST /api/v1/sessions/{id}/control — Stop, restart, recover, or resume a managed session
  • PATCH /api/v1/agents/{id} — Update status + note
  • GET /api/v1/agents/{id}/last-read — Last message agent read
  • POST /api/v1/agents/{id}/heartbeat — Alive signal
  • POST /api/v1/messages/send — Send message (returns recipient status)
  • GET /api/v1/messages/inbox/{id} — Check inbox (supports mode=headers and messageId=...; replies include parent context)
  • POST /api/v1/messages/{id}/read — Mark one message read/unread for its recipient
  • DELETE /api/v1/messages/{id} — Unsend a message
  • POST /api/v1/messages/conversation/clear — Clear a direct chat between two identities
  • POST /api/v1/dispatch — Queue active work for triggerable manual sessions or managed agents
  • GET /api/v1/dispatch/runs/{id} — Inspect a dispatch run, events, and controls
  • POST /api/v1/dispatch/runs/{id}/control — Request interrupt or steer for an active run
  • GET /api/v1/dashboard/dispatches — Dispatch-focused dashboard view
  • GET/POST /api/v1/channels — Channels
  • POST /api/v1/channels/{name}/join — Join or invite agent
  • POST /api/v1/channels/{name}/send — Post to channel
  • GET/POST /api/v1/shared — Shared files (text + binary upload)
  • GET/PUT /api/v1/settings — Settings
