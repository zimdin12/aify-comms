# aify-comms — Codex project notes

Inter-agent communication hub: messaging, channels, file sharing, active dispatch, and a dashboard for Codex, Claude Code, OpenCode, and other MCP-connected coding agents. This file is loaded by Codex when working **on this repo itself**; usage docs for someone installing aify-comms live in [README.md](README.md).

## Primary entry points

- [README.md](README.md) — what the service is, setup, day-to-day usage, API reference.
- [install.codex.md](install.codex.md) / [install.claude.md](install.claude.md) / [install.opencode.md](install.opencode.md) — per-runtime install guides (wrappers, hooks, verification).
- [DECISIONS.md](DECISIONS.md) — rationale for non-obvious design choices and current runtime limits.
- `.agents/skills/aify-comms/SKILL.md` — agent-facing usage guide (tool reference, multi-instance matrix, status table).
- `.agents/skills/aify-comms-debug/SKILL.md` — known issues and fixes (AbsolutePathBuf, hard-reset sequence, buffer_full, orphaned runs, stale bridges).

## Developing on this repo

```bash
git pull
docker compose up -d --build            # rebuilds the Python service container
curl http://localhost:8800/health        # should return {"status":"healthy"}
```

Changes under `service/`, `mcp/`, and `config/` are COPY'd into the container image — rebuild after editing any of them. Changes to docs, skills, `install.sh`, and `.agents/` do not need a container rebuild.

The MCP stdio bridges under `mcp/stdio/` run on the **host**, not in the container. They are loaded by Codex at startup, so changes there require restarting the client wrapper (`codex-aify`) — not a container rebuild.

## Development notes

- **Always point MCP servers at the `server.js` in this repo** — never copy it elsewhere. That's how everyone gets security fixes automatically.
- **Forward-slash `cwd` on Windows.** The bridge auto-normalizes, but any new Codex thread created by the wrapper must be launched from a directory whose path can be rendered forward-slash. On Windows that means launching `codex-aify` from the target project directory, not from home.
- **Re-register is a full state refresh** for everything except `description`. See DECISIONS.md for why and what it means for stale `sessionHandle` and `runtime_state`.
- **Skill files live in two places:** `.agents/skills/aify-comms*/` and `.claude/skills/aify-comms*/`. Keep them in sync when editing.
- **WSL note:** if your Codex CLI lives in WSL, run the installer from WSL too so the registered `cwd` and `codex app-server` paths stay in the same Linux environment. Windows desktop Codex and WSL Codex use different thread stores; resident triggering only works when the bridge talks to the same store as the session.
- **Container name is `aify-comms-service`** on the `aify-comms-network` network. Compose project name is driven by `COMPOSE_PROJECT_NAME` in `.env`.
- **No secrets in commits.** `.env` is gitignored; `config/service.json` is generated by `setup.sh`.

## Testing a change

```bash
# Backend change (service/ or mcp/sse_server.py)
docker compose up -d --build && curl http://localhost:8800/health

# Bridge change (mcp/stdio/)
# Restart codex-aify in whatever session tests the change.
node --check mcp/stdio/server.js
node --check mcp/stdio/runtimes.js

# Python change
python -c "import ast; ast.parse(open('service/routers/api_v2.py').read())"
```

Full end-to-end test is: a two-session dispatch round-trip. Register two agents, `comms_send` from one to the other with `trigger=true`, verify the target wakes and responds.
